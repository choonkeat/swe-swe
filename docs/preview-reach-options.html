App Preview: reach options

Ways to make a session's app(s) reachable in the App Preview tab, and how they trade off. The core problem: the Preview iframe runs in the user's browser, which is usually on a different machine than swe-swe, so localhost / *.lvh.me there point at the user's own loopback, not swe-swe. See ADR-0045 and multi-service.md.

Two axes. Rows 1–4 are addressing forms (how you name a target); rows 5–10 are reach mechanisms (how the browser's DNS gets to swe-swe). You combine one addressing form with one reach: e.g. “named vhost” (form) over “<ip>.sslip.io” (reach).

Recommended spine: ports first

The ten rows below are the full design space, but for the common case — swe-swe on a fixed hostname you don't control DNS for (my-swe-swe.example.com, or localhost) — the whole thing collapses to one idea: a unique port per service is the only identity the proxy needs. The browser varies the port against the fixed hostname; the proxy routes on the port and forwards to the service's loopback address. No wildcard DNS, no fake subdomains, no Host rewriting.

Why port-only is the most robust default (not a compromise). If you don't rewrite Host, the service sees Host: my-swe-swe.example.com:<port> — the same origin the browser is on. So any absolute URL, Location: redirect, or Set-Cookie the app emits references a host:port the browser can actually reach. The proxy is fully transparent and every “fabricated-Host leak” failure mode disappears. Bonus: the browser never leaves the login host (only the port changes, and cookies are port-agnostic on the same host), so password deployments just work — none of the cross-host auth-cookie pain in the note further down applies.

Three decoupled layers

These do not have to agree with each other, which is what makes the model coherent:

LayerValueConsumed byMust be unique per
UI labelapp123 (Procfile service name)the human, in the Preview tabnothing — purely cosmetic
Browser URLmy-swe-swe.example.com:<port>the browser's DNS + the proxy's demux(session, service) → allocate the port per pair
Upstream Hostpassed through untouched (or fabricated only if forced)the service / its in-container routern/a — free string when you must fabricate

Consequence: service names are display-only labels scoped to their session; the (globally-unique) port is the sole identity the proxy routes on. The same name (web, app123) may repeat freely across sessions — different sessions, different ports, no collision. Within one session, duplicate names are rejected at parse time (swe-run refuses to boot: “duplicate service name”), so intra-session identity is guaranteed too.

Three tiers — escalate only when forced

  • default Port per service, Host passthrough. Whenever you control the port surface (VPS, localhost). Cleanest, transparent, no leaks.
  • fallback A Fabricate an upstream Host (e.g. app123.{session}.lvh.me) only for apps whose own router demands a specific vhost name and 404s otherwise. This re-introduces the leak checklist below — but only for those apps.
  • fallback B Single-port path/label demux (/proxy/{uuid}/preview/) when the platform gives you one port only (443-only PaaS like Fly/Railway) so you physically can't open a port per service.
Leak checklist — applies only when you fabricate a Host (fallback A). A fake upstream Host must never escape back into anything the browser acts on, or it resolves to the user's own loopback and dies:
  • Location: redirect headers pointing at the fake host → rewrite back to host:port.
  • Set-Cookie; Domain=<fake host> → browser silently drops it; app sessions/CSRF break.
  • Absolute asset/link URLs baked into HTML using the fake host → fail to resolve.
If the app emits relative URLs and host-agnostic cookies, none of this fires. The moment it hard-codes its Host into responses you're into response-body rewriting — the hacky tier. Port-only (the default) has no fake host, so this checklist is empty for it.

Wildcard + password cookie: option A shipped (2026-07-13). Previously the login cookie was host-only in non-tunnel mode, so wildcard forms loaded a different host (app1-5000.<reach>) that never received it → 401. Now the auth cookie is pinned Domain=<reach> when the browser lands on a server-configured reach (SWE_PREVIEW_VHOST_SUFFIX / SWE_PREVIEW_REACH_DOMAIN, default lvh.me), so one login covers every {name}-{port}.<reach> sibling. Only an auto-derived reach the server is not configured for (e.g. <ip>.sslip.io) stays host-only — set SWE_PREVIEW_REACH_DOMAIN to that domain, or use pinned mode. Mirrors the tunnel fix (ADR-0043 §1).
