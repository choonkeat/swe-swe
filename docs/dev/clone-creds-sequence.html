Credentialed clone reuses the existing HTTPS-cred path

How the proposed New-Session clone (bottom) resolves git credentials through the exact same broker / helper / cred-store machinery that today serves git fetch / git push from a live session (top). The green band is byte-identical between the two.

2 — Proposed: clone a new repo from the New Session dialog

Same rule across all three: apply silently if we have it; only surface fields when the user clicks Change or when auth fails. Public repos trigger none of this.

paste clone URL → Next → parse host, read localStorage swe-swe-creds:<host> → branch by what's stored ↓

FRESH

No PAT stored for this host (fresh install).

User sees

Credential fields appear, empty, host prefilled from the URL, username x-access-token.

What happens

User enters PAT → saved to swe-swe-creds:<host> → sent with prepare → clone succeeds. This is the whole bootstrap — no blank project, no .git/config edit, no reset --hard.

TRANSPARENT

PAT already stored for this host, and valid.

User sees

No fields. Just a quiet line: “Using saved credentials for <host> · Change”.

What happens

Stored PAT is attached to the prepare POST and the clone runs silently — zero extra clicks. Change expands the fields prefilled (masked) for a one-off override (e.g. a different org's token).

REJECTED

PAT stored but the host refused it (expired / wrong scope).

User sees

Fields auto-reveal, prefilled, with “Saved <host> token was rejected — update it?”.

What happens

User fixes just the PAT → localStorage updated → clone retried. No dead-end raw git error.

Once entered, it's entered everywhere

All three states read/write the same swe-swe-creds:<host> store the Session Settings UI uses. So a PAT typed at clone time is instantly available in Settings and to every future clone of that host — and the session created on the cloned repo auto-restores it (existing _maybeAutoConnectSecrets), with the (origin, init_sha) trust pre-seeded from prepare so there's no second “trust this device?” prompt. git fetch/git push inside the new repo then just work.

What is shared vs. new

Everything inside the green band — git → helper → broker (SO_PEERCRED + ancestry walk → sid) → getCredential(sid, host) → PAT → Remote — is the same code path for fetch, push, and clone. The clone flow only adds the four NEW steps that put a credential + a resolvable sid in place before git starts, then tears them down after.
