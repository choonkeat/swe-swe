Diagnosed and fixed it -- and it's not a locale or CLI-flag problem.

Root cause: no color-emoji font in the container

That session's Chromium renders emoji as boxes (aka "tofu" / .notdef) because the container ships only the DejaVu font family and no emoji font at all. When Chromium hits an emoji codepoint and no installed font has a glyph for it, it draws the missing-glyph box.

Evidence from this environment (same swe-swe image):
  • fc-list → 6 fonts, all fonts-dejavu-core. No Noto/emoji/color font.
  • apt list --installed | grep font → only fonts-dejavu-core.
  • Locale is C.UTF-8 (perfectly fine -- UTF-8 aware) and Chromium has no flag that would cause this. So the "wrong locale / wrong flags" hypotheses are both ruled out. It's purely a missing font.

Proof (this ran here, live)

  1. Confirmed baseline: emoji → boxes / mojibake.
  2. Installed one font, no root needed:
   mkdir -p ~/.local/share/fonts
   curl -sSL -o ~/.local/share/fonts/NotoColorEmoji.ttf \
     https://github.com/googlefonts/noto-emoji/raw/main/fonts/NotoColorEmoji.ttf
   fc-cache -f
  1. Re-rendered -- full color emoji (screenshot attached). Notably it even took effect on the already-running Chromium; a browser restart isn't strictly required but is the safe bet if it doesn't.

What a fix looks like

Local / right now (per-session, no sudo -- and we don't have sudo anyway):
The 3 commands above. Fontconfig reads ~/.local/share/fonts per-user, so no root. Add fonts-noto-cjk similarly if you also need Chinese/Japanese/Korean text (that would tofu too).

Upstream in swe-swe (the real fix):
Add emoji (and ideally CJK) fonts to the image that builds the Xvfb+Chromium browser layer. One line in that Dockerfile's apt install:
RUN apt-get update && apt-get install -y --no-install-recommends \
      fonts-noto-color-emoji fonts-noto-cjk \
    && rm -rf /var/lib/apt/lists/*
fonts-noto-color-emoji is the Debian package for exactly the font I downloaded. This fixes it for every session, screenshots, and the noVNC Agent View -- not just per-container patching. It's a well-known gap in slim/headless browser images.
